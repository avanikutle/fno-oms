Mock Data Server and DB Refactor

Documentation Date: July 2026
Scope: Mock Websocket framework implementation and Order Database Normalization.

Overview

The system has been successfully upgraded to support a fully offline, synthetic binary data feed that perfectly mimics the MStock websockets. Additionally, the core Order tables have been refactored to treat every order leg (BUY/SELL) independently with strict auditing capabilities.

1. Database Schema Overhaul

  • Dropped and recreated orders and order_details tables to support the new normalized structure.
  • Added business_date: Inserted as the first column, defaulting to CURRENT_DATE across both tables.
  • Added order_source: New column inserted to distinguish manual vs ALGO broker orders (e.g., 'ALGO', 'BROKER_WEB').
  • Decoupled Legs: Refactored order_details to decouple BUY and SELL legs. Every leg is now inserted as an independent row with its own transaction_type. Columns like exit_order_id and pnl have been removed from the schema.
  • Audit Triggers: Appended updated_at and updated_by columns with automatic PostgreSQL triggers to all major tables: orders, order_details, mstock_scrip_master, scrip_master, quote_ticks, and strategies.

2. Java DAOs and Services Adapted

  • OrderDetailsDAO.java was fundamentally modified. It no longer maps open and exit legs into a single row. Instead, methods like insertEntry and updateExit now insert standalone, distinct rows for every order execution.
  • The primary Order domain entity was updated to supply orderSource = "ALGO". This routes correctly into the refactored schema via OrderDAO.java.

3. Standalone WebSocket Mock Server

A new robust, standalone Mock WebSocket Publisher has been built.

  • Integrated the lightweight Java-WebSocket dependency via pom.xml.
  • Developed MockDataPublisherServer.java on port 8081 to construct and batch encode Mock little-endian binary packets. This perfectly simulates live MStock data streams.
  • Built a standalone launcher MockServerMain.java which operates completely independently of Tomcat.
  • Binary Parsers Abstracted: Abstracted binary MStock parsing and JSON AngelOne parsing logic out of the listeners into a unified utility: BinaryParserUtil.java.
  • Seamless Switch: MstockDataListener was left completely untouched. To utilize the mock setup, you simply modify the websocket URL credential, prompting the current listener logic to flawlessly switch targets to the local mock stream!

4. How To: Setup & Test

Step 1: Start the Mock Server

Since the Mock Server is a completely standalone application, you can start it separately from the main Tomcat instance.

Run the main method inside:

src/main/java/com/fnooms/mock/MockServerMain.java

You should see logs indicating:

[main] INFO MockServerMain - Mock WebSocket Server is listening on ws://localhost:8081

Step 2: Update Application Credentials

You need to tell the Tomcat application to connect to the mock server instead of the live MStock server. Update the websocket URL inside your database:

UPDATE algo_key_value 
SET key_value = 'ws://localhost:8081' 
WHERE key_name = 'creds.mstock.ws.url';

Step 3: Update Feed Broker

Ensure that the application is configured to use the MStock feed (which will now route to your mock server).

UPDATE algo_key_value 
SET key_value = 'MSTOCK' 
WHERE key_name = 'algo.feedBroker';

Step 4: Start the Main Application

Start your Tomcat server (or run Cargo). The MstockDataListener will boot up, read the ws://localhost:8081 URL from the DB, and connect to your standalone Mock Server.

mvn cargo:run -DskipTests
Success! You will now see mock binary packets being ingested, decoded by BinaryParserUtil, and triggering strategies within your logs—entirely offline!
